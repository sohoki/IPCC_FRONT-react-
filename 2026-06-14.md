### 문제
- `WebserverApplication.java`가 `@Bean ServletWebServerFactory`로 HTTP 8081 추가 커넥터를 제공했음
- 파일 삭제 후 프론트엔드 → `http://localhost:8081` 호출 시 `ERR_CONNECTION_REFUSED`

### 수정 내용
`WebServerFactoryCustomizer` 방식으로 대체 (기존 Factory 교체 없이 커넥터만 추가):

```java
@Configuration
public class HttpConnectorConfig {

    @Value("${server.servlet.port}")
    private int httpPort;

    @Bean
    public WebServerFactoryCustomizer<TomcatServletWebServerFactory> httpConnectorCustomizer() {
        return factory -> {
            Connector connector = new Connector("org.apache.coyote.http11.Http11NioProtocol");
            connector.setPort(httpPort);
            factory.addAdditionalTomcatConnectors(connector);
        };
    }
}
```

## 14:00 — JPA UPDATE 미작동 문제 (주요 이슈 — 3차 시도 끝에 해결)

**파일:**
- `backend/src/main/java/egovframework/com/config/EgovConfigAppDatasource.java`
- `backend/src/main/java/com/system/backoffice/bas/program/service/ProgrameInfoManageService.java`

### 문제
`updateProgrmInfo()` 에서 JPA `findById()` 후 엔티티 수정 시 UPDATE SQL이 발생하지 않음.
`SELECT`는 정상 실행되나 수정 내용이 DB에 반영되지 않음.

---

### 실패 이력 1 — DataSourceTransactionManager + dirty checking

```java
// 시도 1: DataSourceTransactionManager 트랜잭션 안에서 dirty checking 기대
@Transactional("transactionManager")
public int updateProgrmInfo(...) {
    ProgrmInfo existing = repository.findById(id).orElse(null);
    existing.updateProgrm(...);  // 필드 수정
    // save() 없이 dirty checking만 기대
    return 1;
}
```

**실패 원인:**
- `DataSourceTransactionManager`는 JDBC 커넥션만 관리
- JPA `EntityManager`의 생명주기를 인식하지 못함
- `findById()` 호출 시 EntityManager가 로컬 트랜잭션으로 별도 실행 → 커밋 후 EM 종료
- 반환된 엔티티는 **detached 상태** → dirty checking 불가

---

### 실패 이력 2 — DataSourceTransactionManager + save()

```java
@Transactional("transactionManager")
public int updateProgrmInfo(...) {
    ProgrmInfo existing = repository.findById(id).orElse(null);
    existing.updateProgrm(...);
    repository.save(existing);  // merge() 시도
    return 1;
}
```

**실패 원인:**
- `save(detachedEntity)` → 내부적으로 `em.merge()` 호출
- `DataSourceTransactionManager` 트랜잭션 안에서 `SharedEntityManagerCreator`는
  JPA 트랜잭션이 없으면 Transient EntityManager 생성
- Transient EM에서 `merge()` 실행 — UPDATE가 발생하지 않음

---

### 실패 이력 3 — JpaTransactionManager + setDataSource() + saveAndFlush()

```java
// EgovConfigAppDatasource.java 수정 시도
@Primary
@Bean(name = "transactionManager")
public PlatformTransactionManager transactionManager(EntityManagerFactory emf) {
    JpaTransactionManager txManager = new JpaTransactionManager(emf);
    txManager.setDataSource(dataSource());  // MyBatis 참여 목적
    return txManager;
}
```

```java
@Transactional("transactionManager")
public int updateProgrmInfo(...) {
    ProgrmInfo existing = repository.findById(id).orElse(null);
    existing.updateProgrm(...);
    repository.saveAndFlush(existing);  // 명시적 flush
    return 1;
}
```

**실패 원인:**
```
InvalidDataAccessApiUsageException: no transaction is in progress
  at SessionImpl.checkTransactionNeededForUpdateOperation
  at SessionImpl.doFlush
```

- `setDataSource()` 를 `JpaTransactionManager`에 설정하면
  Hibernate가 `DataSourceUtils`에서 JDBC 커넥션을 가져옴
- JPA 로컬 트랜잭션(`em.getTransaction()`)이 아닌 JDBC 트랜잭션으로 처리됨
- `flush()` 시점에 Hibernate가 JPA 트랜잭션 활성 여부 확인 → 없음 → 에러

---

### 해결책 — 트랜잭션 매니저 분리 운영

**핵심 아이디어:**
- `DataSourceTransactionManager` (기존) → MyBatis 전용으로 유지
- `JpaTransactionManager` (신규, `setDataSource()` 없음) → JPA write 전용

```java
// EgovConfigAppDatasource.java 추가
@Bean(name = "jpaTransactionManager")
public PlatformTransactionManager jpaTransactionManager(EntityManagerFactory emf) {
    return new JpaTransactionManager(emf);  // setDataSource() 호출 없음
}
```

```java
// ProgrameInfoManageService.java
@Transactional("jpaTransactionManager")  // ← 변경 포인트
public int updateProgrmInfo(ProgrmInfoDto progrmInfoDto) throws Exception {
    try {
        if (Globals.SAVE_MODE_INSERT.equals(progrmInfoDto.getMode())) {
            ProgrmInfo newEntity = ProgrmInfo.builder()
                    .progrmFileNm(progrmInfoDto.getProgrmFileNm())
                    .progrmStrePath(progrmInfoDto.getProgrmStrePath())
                    .progrmKoreanNm(progrmInfoDto.getProgrmKoreannm())
                    .progrmDc(progrmInfoDto.getProgrmDc())
                    .url(progrmInfoDto.getUrl())
                    .build();
            repository.save(newEntity);
        } else {
            ProgrmInfo existing = repository.findById(progrmInfoDto.getProgrmFileNm())
                    .orElseThrow(() -> new IllegalArgumentException(
                        "수정할 프로그램 정보가 없습니다: " + progrmInfoDto.getProgrmFileNm()));
            existing.updateProgrm(
                    progrmInfoDto.getProgrmStrePath(),
                    progrmInfoDto.getProgrmKoreannm(),
                    progrmInfoDto.getProgrmDc(),
                    progrmInfoDto.getUrl()
            );
            // save() 없음 — JpaTransactionManager 커밋 시 dirty checking이 UPDATE 자동 발생
        }
        return 1;
    } catch (Exception e) {
        log.error("updateProgrmInfo error", e);
        return -1;
    }
}
```

### 동작 원리 (성공)

```
jpaTransactionManager 트랜잭션 시작
    ↓
EntityManagerHolder를 현재 스레드에 바인딩
    ↓
findById() → SharedEntityManager가 바인딩된 EM 사용 → managed entity 반환
    ↓
existing.updateProgrm() → managed entity 필드 변경
    ↓
메서드 리턴
    ↓
JpaTransactionManager.commit()
    ↓
Hibernate dirty checking → 변경된 필드 감지
    ↓
UPDATE SQL 자동 실행
```

### 원인 분석

**1. 근본 원인 — `systemCode` 미전달 (400 Bad Request)**

백엔드 `selectIdCheck()` 메서드가 `@RequestParam("systemCode")`를 필수 파라미터로 요구하는데,
기존 호출 코드에서 `extraParams` 없이 호출해 400 에러 발생:

```java
// 백엔드 (EgovCcmCmmnCodeManageController.java)
@GetMapping("/codeIDCheck/{codeId}.do")
public ResultVO selectIdCheck(@PathVariable("codeId") String codeId,
                              @RequestParam("systemCode") String systemCode,  // ← 필수
```

```js
// 수정 전 — systemCode 미전달
await handleIdCheck(form.codeId, setForm);

// 수정 후 — systemCode 포함
await handleIdCheck(form.codeId, setForm, { systemCode: form.systemCode });
```

**2. 구조적 원인 — Swal 이중 호출로 인한 페이지 이동**

`fnAjaxFetch`의 에러 처리 흐름:
```
fnAjaxFetch
  └─ Swal.fire(에러) → (await 없음) → .then(() => /error 이동) 예약
  └─ throw HandledError

use-id-check.js catch
  └─ await Swal.fire() ← 새 Swal이 이전 Swal을 즉시 닫음
                          → .then() 즉시 실행 → /error 이동
                          → 두 번째 Swal은 화면에 뜨지 못함
```

SweetAlert2는 새 인스턴스 호출 시 이전 인스턴스를 닫기 때문에,
`use-id-check.js` catch에서 `await Swal.fire()`를 호출하면 `fnAjaxFetch`의 Swal이 닫히고
예약된 `.then(() => redirect)` 가 즉시 실행됨.

### 수정 내용

```js
// use-id-check.js — HandledError는 재처리하지 않음
} catch (e) {
    if (e?.name !== 'HandledError') {
        await Swal.fire({ icon: 'error', title: '오류', text: e?.message || '중복 체크 중 오류가 발생했습니다.' });
    }
```

## 18:00 — 세부코드 등록/삭제 후 서브 그리드 미갱신 수정

**파일:**
- `frontend/src/pages/Backoffice/Basic/CodeInfo.jsx`
- `frontend/src/pages/Backoffice/Basic/components/CodeDetailCellRenderer.jsx`

### 문제
세부 코드(상세코드) 등록·수정·삭제 후 서브 그리드(AG Grid master-detail)가 갱신되지 않음.

### 원인 분석

`CodeDetailCellRenderer`는 로컬 `rowData` state로 `AppAgGrid`를 제어한다.
기존 `refreshDetailRows`는 `info.api.setGridOption('rowData', rows)` 방식으로 외부에서 그리드 데이터를 교체했는데, **React의 다음 렌더링 사이클에서 컴포넌트가 이전 state를 그대로 내려보내 덮어쓰기** 때문에 갱신이 무효화됐다.

```
refreshDetailRows 호출
  └─ info.api.setGridOption('rowData', newRows)  ← AG Grid에 직접 주입
       └─ React 렌더 발생 시
            └─ CodeDetailCellRenderer가 이전 rowData state를 props로 전달
                 └─ AgGridReact가 이전 데이터로 덮어써짐 (갱신 무효)
```

삭제의 경우 `handleCodeDelete`에 `callback`도 없어 갱신 시도 자체가 없었음.

### 해결 방법

`subRefresh` 카운터를 도입해 React 상태 흐름 안에서 재조회를 트리거한다.

```
등록/수정/삭제 성공
  └─ refreshDetailRows({ codeId })
       └─ setSubRefresh(prev => ({ ...prev, [codeId]: +1 }))
            └─ gridContext 업데이트 (subRefresh 포함)
                 └─ CodeDetailCellRenderer의 useEffect 감지
                      └─ ctxRef.current.fetchDetail() 호출
                           └─ setRowData(newRows)  ← React state 정상 갱신
```

### 수정 내용

**CodeInfo.jsx**

```javascript
// 1. subRefresh state 추가
const [subRefresh, setSubRefresh] = useState({});

// 2. refreshDetailRows 단순화 (data fetching 제거)
const refreshDetailRows = useCallback(({ codeId }) => {
    setSubRefresh(prev => ({ ...prev, [codeId]: (prev[codeId] || 0) + 1 }));
}, []);

// 3. callback에서 await 제거 (sync 함수)
callback: () => { refreshDetailRows({ codeId: detailForm.codeId }); },

// 4. gridContext에 subRefresh 추가
const gridContext = useMemo(() => ({ ..., subRefresh }), [..., subRefresh]);
```

**CodeDetailCellRenderer.jsx**

```javascript
// context 전체 대신 subRefresh[codeId] 카운터만 dep으로 사용
// context는 ref로 항상 최신값 참조
const ctxRef = useRef(context);
useEffect(() => { ctxRef.current = context; });

useEffect(() => {
    ctxRef.current.fetchDetail({ codeId, systemCode, ... })
        .then((list) => setRowData(list.map(...)));
}, [data?.codeId, data?.systemCode, node?.id, context?.subRefresh?.[data?.codeId]]);
//                                             ↑ 이 값이 바뀔 때만 재조회
```
